# 脚本辅助上下文压缩方案

## 1. 目标

用脚本先做机械压缩，减少模型需要阅读和输出的 token。

核心思路：

```text
脚本负责提取、去重、截断、归类。
模型只负责理解、合并、判断重要性。
```

## 2. 能用脚本做的事

### 2.1 提取用户要求

脚本可以从用户消息里提取明显规则。

可识别模式：

- “不要...”
- “我希望...”
- “以后...”
- “记住...”
- “保持...”
- “用最简...”

输出示例：

```json
{
  "userRequirements": [
    "不希望只看到最近20轮对话",
    "所有对话保持最简"
  ]
}
```

### 2.2 记录文件访问

工具调用天然知道读了哪些文件、改了哪些文件。

脚本直接记录：

- 定位到的文件
- 读取过的文件
- 修改过的文件
- 跳过的文件
- 文件 hash 或 mtime

输出示例：

```json
{
  "files": {
    "read": ["docs/Memory管理/Memory管理方案.md"],
    "edited": ["docs/Memory管理/Memory管理方案.md"],
    "skipped": []
  }
}
```

### 2.3 记录命令执行

脚本不保存完整命令输出，只保存摘要字段。

记录内容：

- 命令
- 工作目录
- exit code
- 是否成功
- 输出前几行
- 错误关键行
- 输出是否被截断

输出示例：

```json
{
  "commands": [
    {
      "command": "npm test",
      "cwd": "/repo",
      "exitCode": 1,
      "status": "failed",
      "errorLines": ["Expected 2 tests, received 1"],
      "truncated": true
    }
  ]
}
```

### 2.4 截断工具结果

长输出不能直接进入上下文。

脚本规则：

- 每个工具结果最多保留固定字符数
- 优先保留错误行
- 保留文件路径、行号、异常名
- 成功命令只保留成功状态
- 大段日志只保留首尾和匹配关键字的行

关键字示例：

```text
error
failed
exception
traceback
warning
cannot
not found
expected
actual
```

### 2.5 去重历史信息

脚本可以删除重复内容。

去重对象：

- 重复读取同一文件
- 重复命令输出
- 重复错误日志
- 重复用户要求
- 重复 assistant 总结

去重策略：

```text
相同文件路径 -> 保留最后一次读取记录
相同命令且结果相同 -> 保留最后一次
相同用户要求 -> 保留第一次来源和最后确认时间
```

### 2.6 维护任务状态

脚本可以维护当前任务状态，不必每次让模型总结。

字段：

```json
{
  "goal": "完善 Memory 管理方案",
  "requirements": [],
  "completed": [],
  "inProgress": null,
  "pendingValidation": [],
  "relatedFiles": [],
  "editedFiles": [],
  "blockers": []
}
```

可自动更新：

- 用户新请求 -> 更新 `goal` 或 `requirements`
- 文件被编辑 -> 加入 `editedFiles`
- 测试未运行 -> 加入 `pendingValidation`
- 命令失败 -> 加入 `blockers`
- 命令成功 -> 更新验证状态

### 2.7 生成事件日志

每轮生成结构化事件。

事件示例：

```json
{
  "id": "evt_001",
  "type": "file_edited",
  "target": "docs/Memory管理/Memory管理方案.md",
  "summary": "补充 claude-code 可借鉴设计",
  "messageId": "msg_123",
  "createdAt": "2026-06-09T11:40:00+08:00"
}
```

事件类型：

- `user_request`
- `file_read`
- `file_edited`
- `command_run`
- `test_result`
- `decision`
- `blocker`
- `memory_updated`

## 3. 脚本不能很好完成的事

这些仍然适合交给模型：

- 判断哪个历史决策最重要
- 合并多个相似但不完全相同的用户要求
- 判断用户真实意图
- 总结复杂讨论的因果关系
- 判断失败方案为什么失败
- 把零散事件压成自然语言摘要

所以不要让脚本完全替代模型。

更合理的是：

```text
脚本输出结构化压缩包。
模型基于压缩包生成 rolling_summary。
```

## 4. 推荐压缩包结构

脚本最终输出一个 `context_pack`。

```ts
type ContextPack = {
  sessionId: string;
  range: {
    fromMessageId: string;
    toMessageId: string;
  };
  userRequirements: string[];
  taskState: {
    goal: string | null;
    completed: string[];
    pending: string[];
    blockers: string[];
  };
  files: {
    read: string[];
    edited: string[];
    skipped: Array<{
      path: string;
      reason: string;
    }>;
  };
  commands: Array<{
    command: string;
    exitCode: number | null;
    status: 'success' | 'failed' | 'unknown';
    importantOutput: string[];
  }>;
  decisions: string[];
  openQuestions: string[];
  recentMessages: Array<{
    role: 'user' | 'assistant';
    content: string;
  }>;
};
```

## 5. 压缩流程

### 5.1 每轮对话后

```text
1. 保存原始消息
2. 脚本扫描本轮工具调用
3. 更新 file_index
4. 更新 command_log
5. 更新 event_log
6. 更新 task_state
7. 判断是否需要模型压缩
```

### 5.2 需要模型压缩时

```text
1. 找出未压缩的旧消息范围
2. 脚本生成 context_pack
3. 删除或截断大段工具输出
4. 把 context_pack 发给模型
5. 模型生成 rolling_summary
6. 记录 lastSummarizedMessageId
7. 后续上下文不再发送这段旧消息原文
```

### 5.3 下次请求组装上下文

```text
current user message
+ task_state
+ rolling_summary
+ relevant file_index
+ recent messages
+ current file content
```

## 6. 是否触发模型压缩

不要每轮压缩。

建议触发条件：

```text
当前上下文 token > 70%
或未压缩消息 > 80 条
或工具调用结果累计 > 30k token
或用户明确要求总结/压缩
```

不触发条件：

```text
短会话
刚执行完工具调用但 tool_result 还没被消费
压缩收益小于成本
当前任务马上结束
```

## 7. 模型主动请求压缩

压缩过程可以让模型在合适时机主动发起，但执行权必须在程序手里。

推荐采用双触发：

```text
程序硬触发：token 太高、未压缩消息太多、工具输出太大
模型软触发：模型判断历史上下文开始影响当前任务，主动请求 compact
```

### 7.1 模型怎么请求

模型不要直接修改历史，也不要直接覆盖 memory。

它只能输出一个明确请求：

```json
{
  "type": "REQUEST_COMPACT",
  "reason": "当前上下文过长，旧讨论已经可以压缩为任务状态和摘要",
  "suggestedRange": {
    "beforeMessageId": "msg_120"
  }
}
```

### 7.2 程序负责什么

这里的程序指调用模型的客户端或后端逻辑。

在本项目里可以是：

```text
Electron 主进程
或本地 Node 服务
或负责调用模型 API 的上下文组装层
```

程序负责：

- 判断是否真的需要压缩
- 判断当前是否处于安全压缩点
- 选择压缩哪些消息
- 调用脚本生成 `context_pack.json`
- 再让模型基于 `context_pack.json` 生成 `rolling_summary.md`
- 保存 `compact_boundary.json`
- 下轮请求时注入压缩后的上下文

### 7.3 为什么执行权要在程序手里

模型可以建议压缩，但不能直接决定压缩结果生效。

程序要保证：

- 不删除当前用户问题
- 不删除系统规则
- 不拆散 `tool_use` 和 `tool_result`
- 不丢掉未完成任务状态
- 不把敏感信息写进长期记忆
- 不让模型随便覆盖旧 memory
- 压缩失败时可以回退到原始 transcript

### 7.4 推荐流程

```text
1. 模型输出 REQUEST_COMPACT
2. 程序检查 token、消息边界、工具调用状态
3. 程序允许压缩后，脚本生成 context_pack.json
4. 模型基于 context_pack.json 生成 rolling_summary.md
5. 程序保存 compact_boundary.json
6. 后续请求使用 rolling_summary + task_state + 最近少量消息
```

如果模型没有主动请求，程序仍然可以根据硬触发条件自动压缩。

## 8. 最小可实现版本

第一版只做脚本，不做模型压缩也可以。

MVP：

```text
1. 记录 readFiles
2. 记录 editedFiles
3. 记录 commands
4. 记录 userRequirements
5. 维护 task_state.json
6. 下轮自动注入 task_state + 最近 10 到 20 轮
```

产品内存储建议：

```text
app.getPath('userData')/ai-workstation.db

tables:
  task_state
  file_index
  command_log
  event_log
  context_pack
  rolling_summary
  compact_boundary
```

说明：

- session 记忆是运行时数据，默认不写进项目目录
- 项目已有存储入口是 Electron `app.getPath('userData')`
- 当前数据库文件是 `app.getPath('userData')/ai-workstation.db`
- 脚本的 `--output-dir` 只用于 CLI 开发、调试和测试
- 产品集成时，脚本返回的结构化结果应写入 SQLite 表
- 用户明确选择项目共享记忆时，才写入项目目录

## 9. 进阶版本

第二阶段再做模型压缩。

```text
1. 脚本生成 context_pack.json
2. 主模型基于 context_pack 生成 rolling_summary.md
3. 保存 compact_boundary.json
4. 旧消息只在需要追溯时检索
```

第三阶段做检索。

```text
1. 给 event_log 建关键词索引
2. 给 memory topic 文件建关键词索引
3. 搜不到再查 transcript
4. 最后再考虑向量检索
```

## 10. 推荐实现顺序

1. 先做事件记录脚本
2. 再做 `task_state.json`
3. 再做工具输出截断规则
4. 再做 `context_pack.json`
5. 再接入模型生成 `rolling_summary.md`
6. 最后做关键词检索和向量检索

## 11. 结论

脚本可以显著降低压缩成本。

最推荐的架构：

```text
脚本压缩 80% 噪音。
模型总结 20% 语义。
```

这样比直接让模型阅读全部历史便宜，也比完全不用模型可靠。
