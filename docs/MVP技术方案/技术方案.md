# MVP 技术方案

## 1. 技术目标

实现一个桌面端 MVP：用户选择本地目录，应用展示文件树，用户可右键把文件或文件夹全名插入对话框，并与 AI 进行普通对话。

MVP 明确不读取文件内容，不上传文件，不做文件索引，不做知识库。

当前开发可在 Linux 服务器上进行，但最终实现、打包和验收环境以 Windows 为主。所有路径、目录读取、数据存储和 Electron 打包逻辑都必须按 Windows 兼容方式设计。

## 2. 推荐技术栈

### 2.1 桌面应用

- Electron：提供桌面壳、本地文件系统访问、系统目录选择能力
- Electron Main Process：负责本地能力和持久化
- Electron Renderer Process：承载前端页面
- Electron Preload：暴露安全的 IPC API 给前端
- Windows 作为主要目标运行平台

### 2.2 前端

- React：构建页面和组件
- TypeScript：类型约束
- Vite：前端构建工具
- Zustand：轻量状态管理
- React Router：页面路由
- CSS Modules 或普通 CSS：样式实现

### 2.3 本地服务

- Node.js：运行在 Electron 主进程
- fs/promises：读取目录结构
- path：处理跨平台路径
- ipcMain / ipcRenderer：前后端通信
- 禁止手动拼接 `/` 或 `\\` 路径

### 2.4 AI 调用

- OpenAI 兼容接口
- fetch：发起接口请求
- 环境变量或本地配置保存 API Key

### 2.5 数据存储

MVP 推荐优先使用 SQLite。

- SQLite：保存会话、消息、配置
- better-sqlite3：Node.js SQLite 驱动

如果想更快起步，可先用 JSON 文件，但后续迁移成本更高。

## 5. 前后端通信方式

使用 Electron IPC，不单独启动 HTTP 服务。

原因：

- MVP 更轻量
- 不需要端口管理
- 不需要额外处理本地服务生命周期
- 更适合桌面应用访问本地文件系统

主要 IPC：

- `dialog:selectDirectory`
- `fileTree:list`
- `chat:sendMessage`
- `session:list`
- `session:get`
- `session:create`
- `session:appendMessage`

## 6. 核心数据结构

### 6.1 文件树节点

```ts
type FileTreeNode = {
  id: string;
  name: string;
  path: string;
  type: 'file' | 'directory';
  children?: FileTreeNode[];
  hasChildren?: boolean;
};
```

### 6.2 会话

```ts
type Session = {
  id: string;
  title: string;
  workspacePath: string | null;
  createdAt: string;
  updatedAt: string;
};
```

### 6.3 消息

```ts
type ChatMessage = {
  id: string;
  sessionId: string;
  role: 'user' | 'assistant';
  content: string;
  createdAt: string;
};
```

## 7. MVP 边界

必须保证：

- 目录读取只返回文件名、路径、类型
- 不读取文件内容
- AI 请求只发送用户输入文字和历史消息
- `添加到对话` 只修改输入框文本

## 8. 技术风险

- 大目录一次性加载会卡顿，需要按文件夹懒加载
- Windows、macOS、Linux 路径格式不同
- Windows 中文路径、空格路径、盘符路径需要重点测试
- Electron 权限和安全配置需要避免直接暴露 Node 能力
- AI 接口失败、超时、API Key 错误需要明确提示
- SQLite 打包后路径和权限需要处理
