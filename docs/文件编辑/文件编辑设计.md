# 文件编辑设计

## 1. 目标

让 AI 可以在用户明确授权后帮助修改当前工作区内的文件。

文件编辑能力必须满足：

- 只允许编辑当前已选择工作区内的文件
- 隐藏或敏感路径必须先询问用户
- AI 不能绕过前端和主进程直接写文件
- 写入前必须让用户看到修改内容
- 用户确认后才真正写入磁盘
- 写入前检查文件是否被外部修改

## 2. 核心原则

### 2.1 AI 不直接写文件

AI 只能提出编辑请求或生成修改结果。

真正的文件写入必须由 Electron 主进程执行。

### 2.2 用户确认优先

任何写操作都要经过用户确认。

首版不做自动保存，不做后台静默修改。

### 2.3 工作区边界

所有文件路径必须满足：

- 位于当前工作区内
- 不能通过 `../` 跳出工作区
- 不能编辑未授权目录

### 2.4 敏感路径二次确认

以下路径需要二次确认：

- 以 `.` 开头的隐藏文件或隐藏目录
- `.git`
- `.env`、`.env.local`、`.env.production`
- `credentials.json`
- `secrets.json`
- `id_rsa`
- `id_ed25519`

用户确认后，本次请求可以继续。

## 3. 推荐产品流程

### 3.1 用户引用文件

1. 用户在文件树中右键文件
2. 点击 `添加到对话`
3. 文件出现在右侧引用文件区域
4. 用户输入修改要求

### 3.2 AI 读取引用文件

1. 前端发送用户问题和引用文件列表
2. 主进程校验路径是否在工作区内
3. 主进程读取允许的文本文件内容
4. 主进程把文件内容交给 AI

### 3.3 AI 生成修改方案

首版建议让 AI 返回完整的新文件内容。

后续升级为 unified diff。

### 3.4 用户确认修改

1. 前端展示修改预览
2. 用户点击 `应用修改`
3. 主进程再次校验路径和文件状态
4. 文件未被外部改动时写入
5. 写入成功后刷新文件预览

### 3.5 冲突处理

如果文件在 AI 读取后被外部程序修改：

- 不直接覆盖
- 提示用户文件已变化
- 用户重新读取文件后再生成修改

## 4. MVP 实现范围

首版只做最小闭环：

- 只编辑单个引用文件
- 只支持文本文件
- 最大文件 1MB
- AI 返回完整新内容
- 前端展示原内容和新内容
- 用户确认后覆盖写入
- 写入前检查原内容 hash

首版暂不做：

- 多文件批量修改
- 自动应用 diff
- 二进制文件编辑
- 文件删除
- 文件重命名
- Agent 自动循环执行

## 5. 数据流

```text
用户输入修改要求
  -> 前端发送引用文件列表
  -> 主进程读取文件内容
  -> AI 生成修改结果
  -> 前端展示修改预览
  -> 用户确认
  -> 主进程校验 hash
  -> 主进程写入文件
  -> 前端刷新文件树和预览
```

## 6. 建议接口

### 6.1 读取引用文件

当前已有引用文件读取能力，可继续复用。

### 6.2 生成编辑建议

```ts
type ProposedFileEdit = {
  filePath: string;
  originalHash: string;
  originalContent: string;
  nextContent: string;
  summary: string;
};
```

### 6.3 应用编辑

```ts
type ApplyFileEditParams = {
  workspacePath: string;
  filePath: string;
  expectedOriginalHash: string;
  nextContent: string;
  allowSensitivePaths: boolean;
};

type ApplyFileEditResult = {
  ok: boolean;
  message: string;
  updatedAt: string | null;
};
```

## 7. 主进程校验

写入前必须检查：

- `filePath` 在当前工作区内
- 路径没有跳出工作区
- 文件仍然存在
- 目标是文本文件
- 文件大小在限制内
- 当前文件 hash 等于 `expectedOriginalHash`
- 敏感路径已获得用户确认

## 8. hash 方案

读取文件时计算原始内容 hash。

推荐使用 SHA-256。

写入前重新读取当前文件并计算 hash。

如果 hash 不一致，说明文件已被外部修改，必须拒绝写入。

## 9. UI 建议

首版可以做一个简单编辑预览区：

- 显示文件名
- 显示修改摘要
- 显示原内容和新内容
- 提供 `应用修改` 和 `取消`
- 冲突时显示重新读取提示

后续升级：

- diff 高亮
- 多文件修改列表
- 单个文件逐项确认
- 撤销最近一次修改

## 10. 风险控制

- 默认只读，只有用户确认后才写
- 不允许 AI 自动连续写文件
- 不允许编辑工作区外文件
- 敏感路径必须二次确认
- 写入失败不重试覆盖
- 写入日志记录到会话历史

## 11. 推荐实现顺序

1. 实现 `applyFileEdit` 主进程接口
2. 给引用文件读取结果增加 `originalHash`
3. 让 AI 返回单文件 `nextContent`
4. 前端展示编辑预览
5. 用户确认后调用 `applyFileEdit`
6. 写入成功后刷新文件预览
7. 冲突时提示重新读取
