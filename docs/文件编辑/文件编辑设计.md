# 文件编辑设计

## 1. 目标

让 AI 在用户明确确认后，帮助修改当前工作区内的文件。

首版目标不是做完整 IDE 级文件操作，而是先做一个安全、可用、可回退的最小闭环。

必须满足：

- 只允许处理当前工作区内的文件
- AI 不能直接写磁盘
- 写入前必须展示修改预览
- 用户确认后才真正写入
- 写入前检查文件是否被外部修改
- 敏感路径必须二次确认

## 2. 实现原则

### 2.1 一个底层写入能力

项目里可能有多个保存入口：

- 用户在编辑器中手动保存
- 用户确认 AI 修改后应用

这些入口不应该各自直接写文件。

它们都应该复用同一个主进程安全写入函数，统一做工作区校验、文件类型校验、hash 冲突校验和敏感路径校验。

### 2.2 AI 只生成修改建议

AI 只能生成修改结果，不能直接写文件。

真正写入必须经过：

```text
AI 生成修改
  -> 前端展示预览
  -> 用户确认
  -> 主进程校验
  -> 写入磁盘
```

### 2.3 首版先简单可用

首版采用完整内容替换，不做 diff 应用。

原因：

- 实现量小
- 冲突判断简单
- 用户容易理解
- 能覆盖大部分单文件修改场景

后续再升级为 unified diff、多文件、删除、重命名等能力。

## 3. MVP 范围

首版支持：

- 单个引用文件
- 文本文件
- 文件最大 1MB
- AI 返回完整新内容
- 前端展示原内容和新内容
- 用户确认后覆盖写入
- 写入前检查原始 hash
- 用户有未保存修改时禁止应用 AI 修改
- 写入成功后刷新当前预览

首版不支持：

- 多文件批量修改
- 自动应用 unified diff
- 大文件局部编辑
- 二进制文件编辑
- 文件删除
- 文件重命名
- Agent 自动循环修改文件

## 4. 产品流程

### 4.1 用户引用文件

1. 用户在文件树中选择或右键文件
2. 点击 `添加到对话`
3. 文件出现在引用文件区域
4. 用户输入修改要求

### 4.2 主进程读取文件

1. 前端发送用户问题、工作区路径和引用文件列表
2. 主进程校验文件是否在工作区内
3. 主进程校验文件类型和大小
4. 主进程读取文本内容
5. 主进程计算 `originalHash`
6. 文件内容和 hash 一起交给 AI 流程

### 4.3 AI 生成编辑建议

首版 AI 只生成一个单文件修改建议：

```ts
type ProposedFileEdit = {
  filePath: string;
  originalHash: string;
  originalContent: string;
  nextContent: string;
  summary: string;
};
```

### 4.4 用户确认修改

前端展示：

- 文件路径
- 修改摘要
- 原内容
- 修改后内容
- `应用修改`
- `取消`

用户点击 `应用修改` 后，前端调用 `applyFileEdit`。

如果当前文件在编辑器中有未保存修改，首版直接禁止应用 AI 修改，并提示用户先保存或取消自己的修改。

### 4.5 主进程应用修改

主进程写入前重新校验：

- 工作区路径有效
- 文件在工作区内
- 文件仍然存在
- 文件是允许编辑的文本类型
- 修改后内容不超过 1MB
- 当前文件 hash 等于 `expectedOriginalHash`
- 敏感路径已经二次确认

校验通过后才写入磁盘。

## 5. 接口设计

### 5.1 文本预览结果

当前 `readTextPreview` 建议升级：

```ts
type TextPreviewResult = {
  content: string;
  originalHash: string;
  size: number;
  modifiedAt: string;
};
```

用途：

- 手动编辑保存时做 hash 冲突检查
- AI 编辑时记录原始版本

### 5.2 引用文件读取结果

当前 `ReferencedFileContent` 建议增加 `originalHash`：

```ts
type ReferencedFileContent = {
  name: string;
  path: string;
  status: 'read' | 'skipped';
  content: string | null;
  originalHash: string | null;
  message: string;
};
```

### 5.3 应用 AI 编辑

```ts
type ApplyFileEditParams = {
  workspacePath: string;
  filePath: string;
  expectedOriginalHash: string;
  nextContent: string;
  sensitivePathConfirmed: boolean;
  summary?: string;
};

type ApplyFileEditResult = {
  ok: boolean;
  message: string;
  size: number | null;
  modifiedAt: string | null;
  nextHash: string | null;
};
```

### 5.4 手动保存接口

现有 `saveTextFile` 也应该升级，复用同一套安全校验。

```ts
type SaveTextFileParams = {
  workspacePath: string;
  filePath: string;
  expectedOriginalHash: string;
  content: string;
  sensitivePathConfirmed: boolean;
};
```

## 6. 主进程内部能力

建议抽出这些内部函数：

```ts
function hashTextContent(content: string): string;
function isInsideWorkspace(workspacePath: string, targetPath: string): boolean;
function isSensitivePath(workspacePath: string, targetPath: string): boolean;
async function assertEditableTextFile(filePath: string): Promise<void>;
async function writeEditableTextFile(params: WriteEditableTextFileParams): Promise<WriteEditableTextFileResult>;
```

`saveTextFile` 和 `applyFileEdit` 都调用 `writeEditableTextFile`。

## 7. 冲突处理

读取文件时计算 SHA-256 hash。

写入前重新读取当前文件并计算 hash。

如果当前 hash 不等于 `expectedOriginalHash`：

- 拒绝写入
- 提示文件已被外部修改
- 让用户重新读取文件后再生成修改

如果当前编辑器里有未保存修改：

- 拒绝应用 AI 修改
- 提示用户先处理未保存内容
- 不做自动合并

首版不做自动合并。

首版采用保守策略：只要用户本地有未保存修改，AI 修改不能应用；只要磁盘 hash 不一致，AI 修改不能应用。

## 8. 敏感路径处理

以下路径需要二次确认：

- 以 `.` 开头的隐藏文件或隐藏目录
- `.git`
- `.env`、`.env.local`、`.env.production`
- `credentials.json`
- `secrets.json`
- `id_rsa`
- `id_ed25519`

确认只对本次操作有效。

## 9. 大文件策略

首版规则：

- 超过 1MB 不读取
- 超过 1MB 不保存
- 不做大文件 AI 编辑

后续升级：

- 支持局部读取
- 支持搜索定位片段
- 使用 unified diff 只展示变更区域

## 10. 删除和重命名策略

首版不做删除和重命名。

后续如果支持删除，必须单独设计操作预览：

```text
将删除文件：docs/old.md
文件大小：12 KB
最后修改：2026-06-08 22:10
删除原因：用户明确要求删除旧文档

[确认删除] [取消]
```

删除必须满足：

- 用户明确提出删除
- 文件在工作区内
- 敏感路径二次确认
- 删除前展示文件信息
- 不允许 AI 静默删除

## 11. 后续升级方向

MVP 稳定后再做：

- unified diff 预览
- 多文件修改列表
- 文件创建
- 文件删除
- 文件重命名
- 大文件局部编辑
- 逐文件确认
- 最近一次修改撤销
- 写入日志进入会话历史
