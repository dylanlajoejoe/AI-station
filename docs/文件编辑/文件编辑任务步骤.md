# 文件编辑任务步骤

## 阶段 1：统一安全保存基础

目标：先补齐所有写文件入口共用的安全校验。

1. 在主进程增加 `hashTextContent`，使用 SHA-256 计算文本 hash。
2. 复用或整理 `isInsideWorkspace`，确保路径不能跳出当前工作区。
3. 增加 `isSensitivePath`，识别隐藏文件、隐藏目录、`.env`、密钥文件等敏感路径。
4. 增加 `assertEditableTextFile`，统一校验文件存在、是文件、扩展名允许、大小不超过 1MB。
5. 增加 `writeEditableTextFile`，统一执行工作区校验、敏感路径校验、hash 冲突校验和写入。

验收标准：

- 工作区外文件不能保存
- hash 不一致时不能覆盖
- 敏感路径未确认时不能保存
- 非文本文件不能保存

## 阶段 2：升级现有手动保存

目标：让当前编辑器保存也走统一安全规则。

1. 升级 `readTextPreview` 返回 `originalHash`、`size`、`modifiedAt`。
2. 前端 `FilePreviewTab` 保存 `originalHash`。
3. 升级 `saveTextFile` 参数，增加 `workspacePath`、`expectedOriginalHash`、`sensitivePathConfirmed`。
4. 保存成功后更新 `preview.content`、`originalHash`、`size`、`modifiedAt`。
5. 保存冲突时展示“文件已被外部修改，请重新打开或刷新”。

验收标准：

- `Ctrl+S` 和保存按钮仍可用
- 外部修改文件后再保存会被拒绝
- 保存成功后未保存状态清空

## 阶段 3：引用文件增加 hash

目标：让 AI 编辑建议绑定读取时的文件版本。

1. 给 `ReferencedFileContent` 增加 `originalHash`。
2. `readReferencedFiles` 读取成功时返回 hash。
3. 发送给 AI 的引用文件上下文包含 hash。
4. 前端保存最近一次引用文件读取结果，用于生成编辑预览。

验收标准：

- 引用文件读取成功时有 hash
- 跳过读取的文件 hash 为 `null`

## 阶段 4：AI 单文件编辑建议

目标：实现最小 AI 编辑闭环。

1. 约束首版只允许一个文件操作建议。
2. 让 AI 返回 `ProposedFileEdit`：`operation`、`filePath`、`originalHash`、`nextContent`、`summary`。
3. `update/delete` 的 `filePath` 和 `originalHash` 必须匹配已读取引用文件。
4. `create` 必须提供新文本文件路径和完整内容。
5. 前端检查目标文件是否已有未保存修改。
6. 如果有未保存修改，禁止进入应用流程并提示用户先保存或取消自己的修改。
7. 前端展示修改、创建或删除建议。
8. 删除建议必须提供确认按钮并弹出二次确认。

验收标准：

- 没有引用文件时不能进入 AI 编辑应用流程
- 多个引用文件时提示首版只支持单文件
- 目标文件有未保存修改时不能应用 AI 修改
- 用户能看到修改、创建或删除的操作摘要
- 删除文件前必须询问用户

## 阶段 5：实现 applyFileEdit

目标：用户确认 AI 修改后安全写入。

1. 在 preload 暴露 `applyFileEdit`。
2. 在类型声明中增加 `ApplyFileEditParams` 和 `ApplyFileEditResult`。
3. 主进程实现 `file:applyEdit`。
4. `file:applyEdit` 调用 `writeEditableTextFile`。
5. 应用成功后刷新当前文件预览和文件树信息。
6. 应用失败时展示失败原因。

验收标准：

- 用户确认后才能写入
- hash 冲突时拒绝写入
- 编辑器存在未保存修改时拒绝应用
- 敏感路径需要二次确认
- 删除文件需要二次确认
- 写入成功后 UI 展示最新内容

## 阶段 6：日志和体验补齐

目标：让用户知道发生过什么修改。

1. 应用成功后在会话中记录修改摘要。
2. 记录文件路径、修改摘要、写入时间。
3. 冲突失败时给出明确恢复建议。
4. 统一保存失败的错误文案。

验收标准：

- 会话里能看到 AI 应用了哪个文件修改
- 用户能区分权限错误、冲突错误、文件类型错误
